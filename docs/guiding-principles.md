# kb-engine 基本指导原则

> **版本**: v1.0 | **日期**: 2026-03-31 | **确认**: Boss (Discord B1R1 复盘讨论)
> **性质**: 不可绕过的基本原则，所有设计决策必须对照检查

---

## 原则 1: SOP 的最终使用场景

1. **唯一消费者是 AI Agent**。SOP 只给 PM Agent 或其他 AI Agent 用，不是给人类用的。不考虑"新员工培训"或"决策手册"场景。

2. **PM Agent 的唯一任务是交付客户结果**。生产阶段只做一件事：PM Agent 读取 SOP，完成真实客户需求——从 IND 到 NDA 全流程中 CRO 公司提供的服务或交付物。

3. **能力逐步建立**。不一步到位。Protocol Design 先做好，验证后扩展到 SAP、再到其他分支。

4. **SOP 是 Agent 的操作系统**。PM Agent 应该通过 SOP 和相关文档知道：
   - 面对什么问题该做什么事
   - 什么情况下自己干，什么情况下创建 sub-agent
   - 什么情况下使用什么工具
   - Sub-agent 创建模板的位置（待讨论：是否放在 kb-engine 中）

5. **角色边界清晰**。
   - Learner Agent 是知识库的**贡献者**（在 Amanda/Walt 指导下维护 shared/kb），不是 SOP 的消费者
   - Learner 使用 SOP 的唯一目的是测试修改结果
   - 当前阶段 Learner 只维护 shared/kb，不涉及 kb-engine

---

## 原则 2: 训练程序的目标和边界

1. **训练程序是纯粹的 SOP 优化引擎**。通过模拟实战（AI 设计 Protocol → 对比 FDA 获批方案 → 提取规则）迭代改进 SOP。

2. **训练程序不掺杂业务流程和业务规则**。不包含：客户如何下单、合同如何签署、Amanda 如何排期、D2V 内部审批流程。这些属于业务层，不进 program.md。

3. **评分标准**：10 维度匹配 FDA 获批方案（match/partial/miss），评的是匹配度不是主观质量。

4. **SOP 更新范围**：只追加/修正决策规则。不改评分标准（evaluate.md 固定）。

5. **人工介入**：Batch 5 后和 Batch 20 后各 1 次审核，中间自动执行。架构/流程改进决策由 Boss 在复盘中发起。

6. **不包含**：实际客户交付、真实 IND 提交、Amanda/Walt 的人工审核循环。

---

## 原则 3: Sub-Agent 的唯一目的

**Spawn 独立 Agent 的目的只有一个：优化 Agent 的上下文，保持 LLM 的专注，减少幻觉，提高执行质量。**

- ❌ 绝不是为了节省时间或通过并发提高效率
- ❌ 绝不能因为拆分 sub-agent 而导致关键上下文丢失，从而影响 LLM 的执行结果
- ✅ 只有当一个任务的上下文会干扰另一个任务的执行质量时，才拆分为独立 sub-agent

**推论**：
- 如果把样本量计算拆成 sub-agent，不是因为"并行更快"，而是因为 scipy 公式推导和临床设计思考混在同一个上下文里会互相干扰
- 如果 SOP 说"Biostat 独立计算"，是因为统计计算需要专注的上下文环境，不是因为组织架构要求

---

## 原则 4: 训练引擎与业务逻辑分离

**训练引擎（program.md）不编码 CRO 的组织结构或工作分工。**

- 角色拆分（Clinical / Biostat / Regulatory）是 **SOP 的内容**，不是训练引擎的逻辑
- 训练引擎只负责：spawn Owner Agent → 给 SOP + 题目 → 收答案 → 评分 → 提取规则
- Owner Agent 读 SOP 后自行决定是否 spawn sub-agent、怎么编排——这是 SOP 在训练中被验证的能力之一
