# v4.1 架构多视角评审

> **日期**: 2026-03-31 00:40 | **评审模型**: opus ×5 | **触发**: Boss 要求多领域评审
> **输入**: program.md v4.1 + 修正后两个前提 + Round 1 实例

---

## 评审人 1: AI 系统架构师

### 优势
1. 信息隔离设计正确（Clinical→Judge→Reviewer 三层隔离防止教学测试泄露）
2. Biostat 串行拆分正确（匹配 agent 能力到任务类型）
3. 共识+裁判流水线产生真实学习信号

### 风险
1. **收敛检查未定义且危险**：无单调改进要求、无规则稳定性指标、无最大迭代上限
2. **SOP 更新逻辑是最弱环节**：无规则冲突解决协议、无粒度控制、无依赖图
3. **扩展性是成本问题**：9 分支×20 案例×8 agents = 1440 次 spawn；分支并行会导致 SOP 语义合并冲突

### 建议
1. **Step 6 前增加 Rule Lifecycle Manager**：去重（语义相似度）、冲突检测、范围标注、置信度评分、低置信度修剪
2. **分支顺序执行，案例分批 3-5 个**：用未训练分支做 holdout 验证，建立 train/test split
3. **SOP 结构化为机器可解析格式**：YAML 化规则（ID/scope/confidence/derived_from/contradicted_by），支持程序化冲突检测和修剪

### 元观察
架构擅长**生成**规则，不擅长**维护**规则。生产阶段的反馈回路现在就要规划。

## 评审人 2: 临床运营专家

### 优势
1. 迭代 SOP 优化闭环是真正的创新
2. 10 维度评分对 v1 来说选择得当
3. 领域特定 SOP 分层（core→domains→indications→regulatory）与真实 CRO 知识结构一致

### 盲区
1. **无安全设计维度**：DSMB、停止规则、SAE 报告——FDA 审查每个方案的安全监测，遗漏会触发 clinical hold
2. **训练数据时间偏差**：参考方案设计于 3-10 年前，监管环境已变（剂量优化、DEI、去中心化试验）
3. **无可行性/可执行性检查**：科学完美但运营不可能的方案——入组率、站点能力、对照药供应

### 建议
1. **立即增加 3 个评分维度**：safety_monitoring / dose_rationale / visit_schedule_complexity
2. **实施"设计年份"标准化**：记录方案设计年份（非获批年份），惩罚与过时实践的匹配
3. **构建 5-10 个永不训练的 Red Team 验证集**：含被撤回的加速审批、需要重大修改的方案、无 FDA 指南的适应症

### 关键判断
- 20 案例可做 PoC，不够做生产系统（需 50-80）
- 需要**负面样本**（被 FDA 拒绝或要求修改的方案）
- 系统目前设计的是方案的"科学骨架"，缺少"运营和安全肉体"
- 预计 plateau 在 0.85-0.90（最后 10-15% 是无法规则编码的临床判断）

## 评审人 3: ML/训练系统专家

### 优势
1. 多 Agent 共识+独立盲评是强评估设计（25% 自评通胀检测证明盲评有价值）
2. 分级规则溯源（FDA Guidance > multi-case > single-case）类似荟萃分析的证据权重
3. 整体架构（生成→评估→分类→更新→迭代）是 sound policy improvement loop

### 风险
1. **开卷数据泄露不可控**：ClinicalTrials.gov 包含每个训练案例的答案。无法区分"SOP 驱动的正确"和"搜索驱动的正确"
2. **Append-only 规则累积导致批次内性能退化**：5 轮清理间隔，矛盾/冗余规则创造 Agent 困惑
3. **20 训练案例对目标规则复杂度 marginally sufficient**：71+ 规则/20 案例 = 接近 1:1，泛化差距 12%

### 建议
1. **实施 web-access ablation**：每 5 个案例跑一次闭卷对比，如果闭卷分显著低于开卷 → SOP 没有捕获知识，搜索在做工作。目标：闭卷分在开卷的 0.10 以内
2. **添加维度级收敛追踪**：聚合 match_score 隐藏了哪些维度在 0.90 哪些在 0.40。需要 dimension-level loss curves
3. **ALT 频率作为规则置信度指标**：高 ALT 率 = SOP 不应有刚性规则；低 ALT 率 = SOP 规则校准良好。用 ALT 频率标注现有规则置信度

### 关键判断
- match_score 可能在 case 12-15 信息饱和
- 需要 shuffled ablation 检测顺序依赖
- 需要不同模型家族的 judge 消除 same-model bias

## 评审人 4: 产品经理

### 优势
1. 训练方法论严谨，基于 ground truth（多数 AI 初创公司跳过验证）
2. SOP-as-LLM-input 架构简单正确（无复杂基础设施、无微调依赖、无供应商锁定）
3. 增量分支方法降低构建风险

### 产品风险
1. **SOP 相对裸 LLM 的边际价值未知**：没有做消融研究（有 SOP vs 无 SOP）。如果 delta < 0.05，整个架构是过度设计
2. **规则冲突解决是隐式的**：59 规则可控，200+ 规则时 LLM 遇到矛盾会静默解决（最后看到的规则赢）
3. **训练流水线是手动的，无法扩展到 9 个分支**：当前速度 5 个月

### 建议
1. **立即做消融研究**：20 案例裸 LLM 跑一遍，对比 match rate。一天工作，ROI 最高的分析
2. **立即分离编排和领域 SOP**：在加 branch 2 之前，创建 orchestration/ 层（workflow、tools-registry、escalation-rules）。训练程序永远不碰编排文件
3. **给每条规则加结构化元数据**：Rule ID、trigger conditions（可解析）、evidence links、confidence、version history、conflict declarations。保持 markdown 但机器可解析

### 关键判断
- MVP = Protocol Design + SAP（2 个分支），不需要 9 个
- 竞争壁垒不在 SOP 规则本身（59 条 markdown 竞争对手 10 分钟读完），而在训练流水线 + 验证案例库 + 客户反馈循环
- 混合格式推荐：保持 markdown 可读性 + 加结构化 header（ID/Trigger/Evidence/Confidence/Conflicts）

## 评审人 5: QA/合规专家

### 优势
1. 多层验证架构（3 Agent + 3 Judge + 1 Reviewer）比多数人类 SOP 开发过程检查更多
2. 显式证据溯源 + 置信度层级优于多数我审计过的药企 SOP
3. Append-only 防变异 + Git 版本控制显示成熟的故障模式思维

### 合规风险
1. **🔴 无合格人类签署进入生产 SOP 的规则**：ICH E6(R3) 和 21 CFR 312.23 要求合格人员负责方案设计决策。全 AI 流水线无法承受监管审查——"AI 共识"不是可接受的答案
2. **🔴 无部署后错误检测或 CAPA 流程**：强输入门控但无输出监控。坏规则进入后无检测机制。等同于发布药品但无不良事件报告系统
3. **🟡 将"已批准"与"最优"混淆**：FDA 批准的方案反映谈判妥协，非金标准。学到的规则会编码这些妥协为最佳实践

### 建议
1. **在任何生产使用前实施 Human-in-the-Loop 签署**：每个治疗领域的 SOP 由持证临床医生 + 生物统计师审核。规则到方案的应用必须有人类确认
2. **建立规则健康监测系统**：每条规则跟踪"上次验证日期"，12 个月未重新验证 → 强制审核。跟踪规则到方案的依赖图，坏规则触发下游影响评估
3. **立即添加规则唯一 ID + 负面溯源**：每条规则 `RULE-DERM-042-v3`。不只记录应用了什么规则，还记录考虑并拒绝了什么规则（重建完整决策过程）

### 生产就绪门控
- ❌ 人类专家验证
- ❌ 前瞻性双轨试验（AI vs 人类设计同一研究）
- ❌ 监管法律审查
- ❌ 责任框架
- ❌ 规则覆盖度审计（对照 ICH E6/E8/E9）
- ❌ 故障模式测试
