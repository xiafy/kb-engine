# program.md v5.0 重构设计文档

> **版本**: v1.0 | **日期**: 2026-03-31 | **状态**: 待 Boss 确认
> **触发**: B1R1 复盘 → 指导原则 3&4 确立 → 训练引擎与业务逻辑分离
> **前置文档**: docs/guiding-principles.md（4 条原则）

---

## 一、问题陈述

program.md v4.1 将 CRO 组织结构硬编码在训练引擎中：

```
Step 2 Phase 1: spawn 3 Clinical Agent（硬编码角色）
Step 2 Phase 2: spawn 1 Biostat Agent（硬编码角色）
Step 2b: Clinical 多数票 + Biostat 为准（硬编码合并规则）
```

违反原则 4：角色拆分属于 SOP 内容，不属于训练引擎。
违反原则 3：Phase 2 串行设计的理由是"避免 429"（效率原因），而非上下文优化。

**后果**：
1. 训练引擎无法泛化到其他分支（SAP 分支可能不需要 Biostat，需要 Regulatory）
2. SOP 中的编排能力（"什么时候 spawn sub-agent"）永远不会被训练改进
3. v4.1 实际上在替 SOP 做决定

---

## 二、设计目标

1. program.md 成为**通用训练引擎**——不包含任何 CRO 特定逻辑
2. 角色拆分、编排、工具选择全部由 **SOP + orchestration/** 定义
3. 训练过程本身验证 SOP 的编排指令质量——如果 SOP 没告诉 Owner Agent 要 spawn Biostat，Item 6/7 分数低，差异分析会发现，SOP 会被更新
4. 保持行为变异捕捉能力（2 个 Owner Agent 独立执行）

---

## 三、架构设计

### 3.1 训练引擎角色（program.md v5.0）

program.md 只定义 3 类 Agent：

| 角色 | 数量 | 职责 | 由谁定义 |
|------|------|------|----------|
| **Owner Agent** | 2（并行） | 读 SOP + 题目 → 自行编排 → 输出 10 维度答案 | program.md |
| **Judge Agent** | 3（并行） | consensus answer vs FDA actual → 评分 | program.md |
| **Reviewer Agent** | 1（条件触发） | 审核拟写入 SOP 的规则 | program.md |

**Owner Agent 内部怎么工作**，program.md 不关心。它可能：
- 自己完成全部 10 维度（如果 SOP 没有编排指令）
- spawn Biostat sub-agent 做 Item 6-7（如果 SOP 说样本量需要专门处理）
- spawn 任何 SOP 要求的 sub-agent
- 使用任何工具（web_search, exec, web_fetch）

### 3.2 每轮执行流程（v5.0）

```
Step 0: 加载配置（config.yaml + case-config.json）
Step 1: 读题（question file）
Step 2: spawn 2 个 Owner Agent
  ├─ Owner A: 读 SOP + orchestration/ + 题目 → 自行编排 → 10 维度答案
  └─ Owner B: 同上（独立 session，利用自然随机性）
Step 2b: Helix 共识合并（2 份答案逐维度对比）
Step 2c: Helix 行为变异分析（SOP 一致性 / 违反 / 额外行为）
Step 3: Helix 提取 FDA 实际答案
Step 4: spawn 3 个 Judge Agent（盲评）
Step 4b: Helix 汇总多数票评分
Step 5: Helix 差异归类（KNOW/REG/STAT/ALT/INFO）
Step 5c: Helix 规则溯源
Step 5b: spawn 1 个 Reviewer Agent（条件触发）
Step 6: Helix 写入 SOP
Step 7: Helix 记录结果 + Git
Step 8: 循环/收敛判断
```

### 3.3 Owner Agent 的输入

Owner Agent 收到：

```
1. SOP 文件列表（sop/core/ + sop/domains/ + sop/indications/ + sop/regulatory/）
2. orchestration/ 目录（编排指令、工具注册、升级规则）
3. 题目（question file 全文）
4. 可用资源路径（FDA Guidance 索引、模板目录）
5. 约束：不能查看本题药物的 FDA Review
```

Owner Agent 的 prompt 不包含任何角色拆分指令——它通过读 SOP 和 orchestration/ 自行决定。

### 3.4 共识合并（2 Agent → 1 Consensus）

```
对每个评分维度（10 个）：
  IF 2/2 一致 → 直接采纳 [consensus: 2/2]
  IF 2/2 不同 → 列出两个选项 + 各自理由 [divergence]
    → Helix 基于 SOP 规则选择更合理的选项，标注 [helix-resolved]
```

**[divergence] 的价值**：
- 如果两个 Owner Agent 对 SOP 理解不同 → SOP 措辞有歧义
- 如果一个 spawn 了 sub-agent 另一个没有 → 编排指令不够明确
- 这些都是 SOP 改进信号

### 3.5 行为变异分析（保留，简化）

Helix 在共识合并时记录：
1. **SOP 理解一致性**：两个 Owner 对同一规则的执行是否一致
2. **编排差异**：Owner A spawn 了哪些 sub-agent vs Owner B
3. **工具使用差异**：一个用了 exec 另一个没有
4. **SOP 违反/跳过**：是否有 Owner 忽略了 SOP 规则
5. **SOP 之外的额外行为**：做了 SOP 没要求的事

只记录事实，不做决策判断。

## 四、SOP 编排指令设计

### 4.1 在 sop/core/protocol-design.md 中增加编排部分

在现有 8 步决策流程之后，新增：

```markdown
## Orchestration — 执行编排

### 整体工作流
收到 Protocol 设计任务后：
1. 阅读完整 SOP（core + 适用的 domains/ + indications/ + regulatory/）
2. 阅读 orchestration/ 中的工具注册和升级规则
3. 执行情报收集（搜索同适应症已批准方案、FDA Guidance）
4. 按 Step 1→8 顺序设计方案

### Sub-Agent 使用指引
以下情况应创建独立 sub-agent（目的：优化上下文，保持专注）：

**样本量精确计算**：当需要用统计公式（scipy/R）计算样本量时，
spawn 一个 Biostat sub-agent，给它：primary endpoint 类型、对照臂、
比较结构、randomization ratio。让它专注做数学计算，不混杂临床设计思考。

**情报收集量大**：当需要搜索 >5 个信息源时，可 spawn 一个 Intelligence 
sub-agent 专门收集，汇总后返回给主 Agent。

### 何时不 spawn
- 能在 1-2 次搜索内获取的信息 → 自己搜
- 定性判断（终点选择、对照臂选择）→ 自己做
- 已有 SOP 规则直接适用的 → 按规则执行，不需要额外 Agent
```

### 4.2 orchestration/ 目录内容（初版）

**orchestration/protocol-design-workflow.md**：

```markdown
# Protocol Design 工作流

## 输入
- 药物信息（名称、MOA、剂型、适应症、目标人群、治疗线）
- 监管路径（NDA/BLA/505(b)(2)）

## 输出
- 10 维度完整设计方案

## 执行步骤
1. 加载 SOP：core/protocol-design.md + 匹配的 domains/ + indications/ + regulatory/
2. 情报收集：搜索同适应症 Phase 3 方案（ClinicalTrials.gov + FDA Drugs@FDA）
3. 按 SOP Step 1→8 逐步设计
4. 样本量计算：spawn Biostat sub-agent（见 SOP 编排指引）
5. 质量自审：对照 SOP 逐条检查
6. 输出 10 维度答案
```

**orchestration/tools-registry.md**：

```markdown
# 可用工具注册表

| 工具 | 用途 | 何时用 |
|------|------|--------|
| web_search | 搜索公开信息 | 情报收集、竞品分析 |
| web_fetch | 抓取网页内容 | 已知 URL 的文档获取 |
| exec | 运行 Python/R 脚本 | 样本量计算、统计分析 |
| read | 读取本地文件 | SOP、FDA Guidance、模板 |
```

**orchestration/escalation-rules.md**：

```markdown
# 升级规则

## 何时停下来
- SOP 中没有任何规则覆盖当前决策点 → 标注 [SOP-GAP] 继续
- 多条 SOP 规则矛盾 → 标注 [SOP-CONFLICT] 选择更高置信度的规则
- 题目信息不足以做出判断 → 标注 [INFO-MISSING] 给出最合理假设
```

## 九、风险与缓解

| 风险 | 影响 | 缓解 |
|------|------|------|
| Owner Agent 不读 SOP 编排指令 → 不 spawn Biostat → Item 6/7 低分 | 首轮可能 | 这正是训练的目的——低分触发 SOP 更新，加强编排指令 |
| 2 Agent 共识不如 3 Agent 稳定 | 分歧增多 | Helix 解决分歧 + 行为变异记录提供更丰富信号 |
| orchestration/ 内容初版不完善 | Owner 行为不一致 | 通过训练轮次迭代改进 |
| config.yaml 扩充后向后不兼容 | 旧 round 记录格式不同 | results.tsv 加版本标注 |

## 十、成功标准

v5.0 重构完成后，用 Round 2（Orgovyx）验证：
1. Owner Agent 能读 SOP 自行完成 10 维度设计
2. 至少 1 个 Owner Agent 自主 spawn Biostat sub-agent
3. match_score 不低于 v4.1 Round 1（0.70）的 baseline
4. 行为变异分析能捕捉到两个 Owner 的编排差异
